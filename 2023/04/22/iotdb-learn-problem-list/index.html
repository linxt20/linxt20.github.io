iotdb学习踩坑记录

发表于2023-04-22|更新于2023-05-09|技术iotdb
|字数总计:1.9k|阅读时长:6分钟|阅读量:

从初学iotdb到现在，在iotdb官网教程的学习当中踩到的一些坑和对应的解决方法，有些问题会非常简单，有些问题可能已经在新版本中被解决了。先注明作者的古董电脑是windows系统

问题集锦

  • 问题1：在官网的教程中有一个使用cli工具的部分，运行提供的指令，告知服务未打开

    • 解决方法: 其实这个部分是一个流程，可能是因为上一个小标题启动iotdb当中没有出现到服务器这个词，以至于还是萌新的我都没发现需要交互的对象是前一步进行操作的结果。所以这里在运行cli工具部分指令前需要先运行start-standalone.bat

  • 问题2：在启动iotdb当中，执行启动iotdb服务器指令出现堆分配内存不够的情况

    • 解决方法: 这是因为我的古董电脑的内存比较小，而iotdb的服务器的配置要求设置的相对较高，只需要把config当中的datanode-env.bat第63、64行从原来的/2修改为/10还有confignode-env.bat第63、64行从原来的/2修改为/10，这里的原理就是在配置上降低对启动iotdb的要求

  • 问题3：在java原生接口中，其中安装方法提供的指令运行报错，找不到程序

    • 解决方法: 在官网的文档当中并没有标注这里是哪里的根目录，这里有待完善，这里应该是指从iotdb的github上pull下来的iotdb源码的文件夹目录，而不是打包好的二进制程序文件夹目录

  • 问题4：在下载与安装当中，提供了编译iotdb源码的指令，运行这个指令mvn clean package -pl server -am -Dmaven.test.skip=true没法顺利实现编译

    • 解决方法：这里的后缀需要修改，最新版的后缀应该是-DskipTests，完整的指令是mvn clean package -pl server -am -DskipTests,这个指令的作用是编译server模块，如果想要编译整个项目可以使用下面的指令mvn clean package -pl distribution -am -DskipTests

  • 问题5：从iotdb的github上pull最新代码master分支，执行编译指令没法跑完，报错显示server、openapi等包没法安装

    • 解决方法: 这个是开发分支，代码还在修改测试，切换到已经发布的rel/1.1的分支即可

  • 问题6：在Java原生接口中，其中在MAVEN中使用接口没有写出maven依赖需要写到的哪里

    • 解决方法: 这里查阅了相关资料，作者当时还是第一次下载了intellj idea，才知道了JAVA的项目有一个pom.xml文件，这部分依赖需要写在pom.xml当中

  • 问题7：用intellj idea创建java项目，发现没有pom.xml文件

    • 解决方法: 使用intellj idea创建java项目的时候需要包管理需要选择maven，这是因为maven的包管理是采用pom.xml文件进行记录的

  • 问题8：在pom.xml中导入依赖之后保存，重新运行项目，发现依赖并没有安装上，import的iotdb包都显示无法使用

    • 解决方法: 这是因为在pom.xml中导入依赖后，需要在点击intellj idea右侧的maven打开maven包管理的小窗口，点击小窗口左上角的reload，才算是成功导入iotdb的依赖包

  • 问题9：在有创建数据库的程序中，反复执行发现数据库的数据一直不变

    • 解决方法: 因为第一次执行就已经创建数据库并写入了数据，后续的操作都是对第一次生成的数据库的覆盖。后续执行前需要先删除数据库再重新执行。

  • 问题10：在集群部署的时候，将已经快速启动过的同一个项目文件夹复制到三个文件夹下，测试在同一台设备上的集群部署，发现只有一个能够能够正常运行

    • 解决方法: 这是因为部署执行一次之后，在项目的data文件夹下会记录项目的部署信息，复制了三份的项目的记录都是一样的，但是集群部署会修改端口，这与部署记录不同，所以没法正常运行，解决方法是把data文件夹下的节点配置文件删除，重新部署即可

  • 问题11：使用python原生接口时，讲官网原生接口的session example copy下来运行发现运行不通过，显示session方法和构造函数等的不存在

    • 解决方法: 在python原生接口的页面当中，通过session example链接跳转到的应该是master分支的参考代码，master分支是目前正在开发的最新版本的代码，与目前发行稳定版本的iotdb的python包存在版本差异，部分函数没有支持，即使copy下来也没法运行，因此在学习session example的时候，记得切换session example所在的github分支与iotdb的python包的版本一致。检查iotdb版本包的指令为:”pip show apache-iotdb”.

  • 问题12：在上一个问题为基础，打开iotdb的服务端，同时运行session example代码，报错: RuntimeError: ('Could not set time zone because: ', TTransportException('Transport not open'))

    • 解决方法: 此时运行的iotdb服务端的版本和iotdb的python包的版本不一致，需要在iotdb的github上pull 与iotdb的python包版本一致的分支内容，随后运行confignode和datanode就行，不用运行二进制项目包当中的客户端，运行python代码连接iotdb的时候，记得先运行iotdb的服务器，但是在开启iotdb服务器之前，需要提前检查自己所使用的iotdb源码(or 二进制项目)的版本与python包的版本是否一致，如果不一致需要去iotdb官网下载对应版本的iotdb源码(or 二进制项目)

  • 问题13：运行iotdb的客户端报错：此时不应有 \Java\jdk1.8.0_351\lib。问题背景是此时正在配置superset的环境

    • 我后来去开启bin下的服务器和客户端是正常的，但是这个时候我的intellj idea崩了，下载了个新的idea，后这个问题就没了，有点玄学
      • 周学长认为这个问题的解决可能可以考虑的方向是卸载java再安装

  • 问题14：我的intellj idea崩了，重新下载之后，重新编译iodtb项目之后，遇到了一个MLNodeClient.java的文件显示了报错，报错是需要的和提供的类不一样(Required type:org.apache.thrift.protocol.TProtoco // Provided:org.apache.thrift.protocol.TProtocol...)，但是实际上显示的类名是一样的，有点玄学(confignode和datanode都是能够正常运行)

    • 解决方法：可能是上一次idea对iotdb的编译缓存了一些索引，和这次编译后的有点冲突，点击左上角文件，选择清除缓存，问题就解决了

  • 问题15：在测试superset当中，由于iotbd的python包中部分包版本与superset产生冲突，需要修改后重新打包，但是进入到client-py中，输入python3 setup.py sdist bdist_wheel没有报错，没有输出，也没有生成文件

    • 解决方法，这个是因为缺少了相关的库文件，但是没有报错确实是很奇怪，这里通过以下指令安装setuptools和wheel包pip install setuptools wheel即可解决
