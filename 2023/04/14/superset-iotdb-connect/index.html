为superset添加iotdb连接器的实现

发表于2023-04-14|更新于2023-05-12|iotdb连接器
|字数总计:3.9k|阅读时长:12分钟|阅读量:

问题背景

陈学长最近在做iotdb和superset的连接工作，实现效果大概是，对于superset前端接收到的操作，会传递给superset后端处理，superset随后将数据请求发送给iotdb的python客户端，python客户端与iotdb服务端交接数据后传输回superset后端，最后展现在superset前端上。其中superset与iotdb的python客户端之间的连接，参考superset官网对于与其连接数据库的要求——满足DBapi规范、sqlalchemy协议以及在superset中重载相应数据库引擎规范(也称为连接器)，相对应的，这个连接需要完成三个模块，其中DBapi和sqlalchemy两个模块陈学长已经完成，我主要是负责在前面工作的基础上，完成在superset中重载相应数据库引擎规范的部分内容。

主要参考资料

本次工作的参考资料部分由陈学长提供，部分是由我在网络上所搜寻所得
- 陈学长完成的关于连接模块联系与初步技术设计的ppt和飞书文档
- iotdb关于python原生接口的使用,包含基础方法使用与iotdb服务端连接，dbapi和sqlalchemy相关知识
- superset官方网站 提供了比较丰富的信息，官方的连接流程和相关信息都可以从这里获得
- superset的github 所提供的是，Superset/db_engine_specs文件夹下包含有引擎规范的基类base.py和其他各种数据库的基于base.py引擎规范重载实现。
- superset调用crateDB的使用指南博客，这里是讲解在代码完成之后，如何用命令行操作完成crateDB和superset的连接与使用
- superset与crateDB联动的基础引擎规范实现直播，则是以crateDB数据库为例讲述了操作规范的缘由与从零到一个基础操作规范的实现流程。
- stack overflow的superset问答区 有价值的信息比较少
- stack overflow pro的superset问答频道 可以回答部分简单的小bug，较为复杂困难的bug还是需要自己完成
- 用pycharm调试superset的在windows上使用pycharm调试superset的配置

整个实现流程规划(实际实现的时候对这部分继续了修改)

通过前面的资料，大概规划了连接器的一个实现流程

  • 1、学习iotdb的python原生接口(包含基础方法使用，dbapi和sqlalchemy，实现连接iotdb)
  • 2、启动superset的开发服务器，实现初步连接
  • 3、参考其他的数据库引擎规范完成iotdb的重载base.py，遵循在使用当中发现问题解决问题的方式，逐一补充重载函数
  • 4、连接器整体测试文件与文档
  • 5、尝试使用docker去运行superset，补充文档

实现流程

学习iotdb的python原生接口

这个部分主要是记录一下作者自己的理解。

  • 在iotdb官网文档当中，关于python原生接口，有一个session example的指导性代码可以参考学习，这里可能会遇到一些错误，可以参考博客——iotdb学习踩坑记录

  • iotdb-dbapi和iotdb-sqlalchemy是封装在iotdb源码当中，其中iotdb-dbapi是一种python的接口规范，通过python获取iotdb服务将会满足这种规范，iotdb-sqlalchemy则是一种连接协议，这里是superset将会采取sqlalchemy协议与iotdb之间进行连接

  • superset的后端通过sqlalchemy直接链接iotdb服务端，superset对数据库的调用指令通过sqlalchemy转化为DBAPI对于iotdb的调用方式，类似于python与iotdb的连接，但是python客户端与iotdb的连接是直接用iotdb的DBAPI，superset通过sqlalchemy去实现基于DBAPI的更高级的操作，一个基于SQLAlchemy的数据库访问层，该层在底层使用IoTDB的DB-API实现。这种结构允许Superset利用SQLAlchemy的高级特性（如连接池、事务处理等），并实现与其他数据库类型的兼容性。5、解决前端崩溃问题，程序顺利跑起来

安装与启动superset，建立superset与iotdb的连接

  • superset的安装与启动的这部分内容属于是相当曲折，读者可以看我的另外一篇博客——superset的安装与启动

  • 对于superset和iotdb的连接的步骤如下

    • 打开superset的本地网站，地址127.0.0.1:8080
    • 按照前面创建的账号密码进行登录
    • 点击右上角头像，选择database connect，进入页面点击+database
    • 在其他数据库当中选择iotdb，如果没有则选择其他，然后输入iotdb的名字
    • URL输入iotdb://root:root@127.0.0.1:6667，点击 test connect，显示good弹窗即可点击连接

重载base.py实现适配iotdb的引擎规范

  • 简单理解：

    • 需要添加的内容在哪里，就是在superset/db_engine_specs当中添加一个iotdb.py即可，可能以前的版本还需要在同目录下的init.py进行注册，但是我阅读了现在的最新版本的init.py文件，里面有一个load_engine_specs的函数，这个函数的执行可以直接添加superset/db_engine_specs当中的所有引擎规范，并不需要自己额外在其中添加代码
    • 需要添加的内容是什么，其实就是对base.py文件的一个重载，我简单浏览了一下base.py文件，发现其和iotdb的sqlalchemy的实现居然有很多相似之处，我大概是理解这部分就是使用sqlalchemy进行通信的关键代码，后续的调试也可以在这里进行

  • 报错1：sqllab当中superset获取不到iotdb的tables列表

    • 这个问题的解决历时比较长，因为一开始通过查阅日志当中的报错信息去查找相关资料
      • superset的github的issue, 关键词搜索”tables”,”SEE TABLE SCHEMA”,”There was an error loading the tables”,”/api/v1/database/1/tables/?q=(force: !t,s”,”422”，和最新的百条
      • superset的slack社区，debugger-together的channel中，关键词检索”tables”,”SEE TABLE SCHEMA”,”There was an error loading the tables”
      • superset的slack社区的提问，社区人员没有回复
      • 咨询chatgpt获取的相关回答
      • 尝试对于base.py文件函数进行输出调试和修改
      • 在iotdb的sqlalchemy范围文件用客户端进行信息获取，输出调试等
    • 解决方式：添加view重载，这个通过pycharm的调试查找到获取不到table的实际错误信息和日志输出是有不小差距的，调试报错指向了get_view_names函数，在iotdb的引擎规范当中重载get_view_names函数即可

  • 报错2：在superset当中选择了table后显示抓取不到table tree mata

    • 解决方法：这里报错信息指向了sqlalchemy中的iotdbsqlcompiler中的问题，把visit_select函数注释掉后再次打包安装即可解决

  • 报错3：换用新的iotdb包后报错缺少模块，无法获取到iotdb的schema

    • 解决方法：这个目前的错误原因目前还没法确定，但是有两个方向，一个是我当时刚刚装好pycharm没有重启而带来的问题，另一个是superset.db留存了上一个包的部分信息产生冲突，因此删除superset.db然后重启电脑，再重启superset

  • 问题4：无法单独查询时间

    • 解决方法：修改了先前被注释的visit_select，因为这个部分的代码是与sqlalchemy的版本关系较强，原先的代码能够运行，但是sqlalchemy更新之后，这个函数有所变化，所以iotdb的sqlalchemy的visit_select也需要调整

  • 报错5：创建dataset的时候，table已经选择好了，但是点击后弹窗显示无法创建，后端报错UNIQUE constraint failed: tables.table_name

    • 解决方法：这里是因为将要创建的table名和已经有的table名重复了，这里需要注意，schema名不同但是table相同也是不能创建的，要么删了原来的，要么现在的table换个名字

  • 问题6：时间戳timestamp被识别为了number

    • 解决方法：这里是superset自身的设置问题，superset的时间戳列需要在dataset的edit当中单独点击设置，需要在列的tab里面给时间戳列点击is temporal 才能被标记为时间

  • 问题7：在选择线性图标时候发现横坐标的时间粒度缺失

    • 解决方法：在iotdb.py当中补充_time_grain_expressions字典的

  • 问题8：从iotdb获取的时间戳显示出现问题，很可能是数量级的转化出现错误

    • 解决方法：根据crateDB在油管的直播视频，他们里面有这个问题的处理方式，需要在iotdb.py中重载了alter_new_orm_column、convert_dttm、epoch_to_dttm、epoch_ms_to_dttm四个函数，然后在superset中给dataset中设置DATETIME FORMAT为epoch_ms。”epoch_ms”是一个常见的时间戳格式，它表示从1970年1月1日00:00:00 UTC开始的毫秒数。

1、聚合函数部分的实现

  • 可能的解决方式：参考视频和官网

    • superset中提供但iotdb中没有的聚合函数 COUNT_DISTINCT
    • 在聚合函数的参数中，不能选择时间和*
  • 目前新方向
    • 找到aggregate的来历，可以直接照着iotdb的聚合函数修改

小bug记录

  • 打开superset网站后，点击sqllab后，报错，后端没法添加新的tab
    • 解决方法：这是因为superset当中没有任何数据库，需要先添加数据库连接后才能正常使用

    • 可能的解决方式，1、重新pull一个superset，再次安装启动，2、清空iotdb的data文件夹里面的confignode和datanode的两个文件夹，重新启动iotdb 3、在2的基础上使用下面的代码创建数据库、创建时间序列、插入数据

吐槽

  • 1、在一开始是找到了一份前人的关于iotdb的引擎规范的参考,但是我仔细一看，发现这就是抄了crateDB的引擎规范，虽然类里面的内容的位置换了一下位置，但是这个类的名称都没改是否有点抽象？？
  • 2、superset前端崩溃问题在issue当中挂了三年左右，主版本依旧没有更新解决方式，确实是比较抽象
  • 3、信了chatgpt的鬼话，缩减了superset的开发环境的安装流程，害我多debug了两天，我属实是急了
  • 4、其实前端崩溃部分只是少了对于前端的编译操作，我在测试2.1版本的时候就想到了这个问题，但是在前端一直编译不成功，但是查了github的issue才发现，2.1版本的这个前端编译是有报错的。其他版本反而正常
  • 5、在superset的slack里面对于部分问题选择性回答还是比较麻烦的，对于table获取不到的问题，搜索了一下基本上没有官方人员回复，在github的issue也是一样的情况

杂谈——离奇曲折的安装经历

  • 萌新作者一开始进行安装还是在本地环境下进行的，后面遭遇到了修改文件来解决问题就有点不知所措，因为本地环境下的依赖安装位置并不好找。后来阅读了一些网络上的博客文章，大多都是采用虚拟环境进行操作，虚拟环境既能做到环境和本地分离，还能共用本地的文件，确实是方便了不少，于是决定改用虚拟环境。后面遭遇了flask_app环境变量的设置问题，查阅的资料都是采用linux的export的方式，对于windows确实是有点点不大友好，于是专门搞了个ubuntu来操作一下superset的安装流程，但是因为虚拟机的网络和下载源的问题，依赖下载不完全，还是回到了虚拟环境进行debug。不过还好最后还是有惊无险的顺利完成安装与启动工作，这个部分花了三个多小时。

  • 第二天准备进行下一步操作的时候发现，我安装和启动superset居然和superset的源码没有沾上关系，我也是被自己离谱到了。陈学长提到了这是生产环境下的版本，与我们进行开发所需要的版本不一样。于是pull了github源码，重新进行安装与启动，一开始是纠结于选用superset的什么分支，选了几个版本都是报错重重，最后还是用master分支。这里面遇到的了一些新的问题，包括了一个不可复现的前端崩溃问题，后面突然莫名其妙好了。这里大约花费了六个小时。

  • 第三天交流iotdb与superset连接问题的时候，superset启动又遭遇了前面的不可复现的前端崩溃。这个问题真的是相当的棘手，当然是在当时看来，现在看来真是自己没有好好读文档遭的罪。不过作者在这里也学会了如何在github中翻找issue，在readme.md中仔细的获取信息，在官方社区中查找解决方式等，首先的是，作者在issue当中了解到很多使用docker进行安装和启动的人也遭遇到了相似的问题，再加上docker更加完善的隔离机制，我也打算改用docker。不过docker才刚刚学习了一些基础，配置好了基础环境，在superset社区的热心群众给我指出了我没有编译前端的问题。我一开始还是不大相信的，因为我在2.1分支当中是进行过前端的编译的，不过遭遇到了一些报错，所以没有进一步的尝试。于是我在master分支上完成了前端的编译工作，最终重新启动了服务器。也就在重启后，我在github的最新的issue，就是当天的issue，发现有人提出来:在其他版本都能够正常编译前端，在2.1版本遭遇到了批量报错问题？？？确实是被离谱到了。不过也有个问题，我前面没有编译前端，前端为什么会出现能够顺利运行的现象？这个对后面问题的处理也造成了一些误导(目前的理解是，因为一开始运行了生产环境的，有前端的缓存，后面出现一直前端崩溃之前是卸载了iotdb的python库带走了部分依赖，当然也有可能是本地编译的时候生成的superset.db存储了前端的信息？)。这个过程大约花费了15小时
